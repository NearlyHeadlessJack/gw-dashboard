# 项目名 gw-dashboard

## 项目简介
gw指星网星座（又叫国网星座，GW星座），是中国正在建设的一个对标星链的卫星互联网星座工程。

本项目用于基于互联网，收集开源的星网运行、发射的信息，存储进数据库进行分析，并用仪表盘、地图来展示运行状态。

## 技术栈
- **前后端分离**
- 后端语言 python>=3.12
- 后端架构
- 前端使用 react架构
- 后端使用FastAPI 和 SQLAlchemy
- 后端可以提供三种数据库 sqlite3 mysql和pgsql的支持
- uv进行包管理
- ruff进行检查和格式化
- pytest进行测试

## 文件结构
- gw 后端源代码文件夹
- frontend 前端代码文件夹
- tests 后端测试代码文件夹
- docs 文档文件夹
- database 用于存放数据库
- tmp 用于存放部分代码原型，并在其中进行测试运行

## 后端程序结构（指gw文件夹里面）
- 入口程序 用于加载配置、初始化数据库、测试网络等
- utils 工具函数库
- database 数据库相关库
- scraper 爬虫相关处理库
- web 前端请求处理库
- daemon 守护进程，用于控制定时爬取和处理数据

## 后端逻辑
- 先启动入口程序，入口程序加载配置、初始化数据库、测试网络，然后启动守护进程，
- 守护进程检查数据库更新日期是否有效，有效则启动web服务器，否则进行第一次更新处理（即爬取、分析、存储数据）。
- web服务器只负责从数据库读取数据并提供给前端，并且支持缓存
- 守护进程会定期更新数据库

## 前端布局
- 交互式、响应式布局，适配大屏和小屏
- 只有两个tab页面，上面是导航条。
- 一个页面是仪表盘，总览数据，左边是菜单
- 通过左边菜单可以切换不同仪表盘页面，日后再补充
- 另一个tab页面是地图，上面显示卫星位置和轨迹

## 当前实现记忆

### 启动与打包
- 程序入口支持 `uv run -m gw` 和 PyPI 安装后的 `gw-dashboard` 命令。
- 默认不需要配置文件；默认使用 SQLite，数据库路径为 `~/.gwtracking/database.db`。
- 后端启动后同时托管前端页面，控制台会打印前端入口 URL。
- PyPI wheel 内置前端静态资源，默认目录是 `gw/web/static`。普通用户 `pip install gw-dashboard` 后不需要 Node.js。
- `frontend/vite.config.ts` 的构建输出目录是 `../gw/web/static`，发布前需要先执行 `cd frontend && npm run build`。
- 启动参数 `-d` / `--build-frontend` 会在启动后端前执行一次 `npm run build`，只适合源码开发或发布前使用，需要本机有 Node.js/npm 和 `frontend` 源码目录。
- 后端仍保留 `frontend.dist_dir` / `GW_FRONTEND_DIST_DIR` 覆盖能力，用于开发或特殊部署。

### 数据更新与服务状态
- 首次运行如果数据库不存在或未初始化，应先爬取并更新数据，再启动 Web 服务。
- daemon 负责定期检查数据有效期并触发更新。
- 服务器状态页面显示上次爬取时间，并提供数据有效期输入框，单位是小时，范围 1 到 48。
- TLE 入库时，如果单颗卫星的新 TLE 历元和数据库现有记录一致，不再重复添加。
- 后端向前端返回制造商名称时，包含“五院”的值统一显示为“航天五院”。

### 前端页面
- 顶部通知栏文案为“数据均来自公开信息，仅供学习参考。”，带 GitHub 项目主页入口和关闭动画；每次重新打开网页都应出现。
- 总览页包含统计卡片、所有卫星列表、所有发射列表、制造商/火箭统计和小地图。
- 总览小地图只显示卫星点，不显示轨迹，用户不能拖动或缩放。视角会随容器大小自适应，必须保证中国始终在可视范围内。
- 小地图和大地图里的卫星当前位置都由前端根据当前时间和 TLE 实时解算。
- 前端当前时间通过后端 `/api/time` 获取，后端时间源使用 `ntp2.aliyun.com`。
- 组与单星页面下拉列表按从近到远排序；可以查看整组或单星信息。
- 历史轨道页面近地点/远地点图表坐标轴自适应，鼠标悬停时高亮当前时间点并显示标签。

### 地图规则
- 地图使用公共高德标准瓦片，不需要 Key，不使用暗黑版。
- Leaflet 默认 attribution flag 已隐藏或替换处理，不展示乌克兰国旗。
- 大地图默认显示所有卫星位置，不默认显示轨迹。鼠标悬停到某颗卫星时，才显示该卫星及同组其他卫星的轨迹和标签。
- 大地图有 `LEO覆盖范围` 开关，打开后显示低轨卫星覆盖范围。
- 覆盖范围计算规则：地面最小仰角 30°，只显示非 GEO、非高轨、轨道倾角不大于 83° 的卫星。GEO、高轨、倾角大于 83° 的卫星不显示覆盖范围。
- 组与单星页面地图也使用同一套覆盖范围过滤规则；选择组时显示整组，选择单星时只显示单星。

### 发射与火箭统计
- 发射统计页面“所有发射”使用独立滚动区域，并显示发射时间、轨道、发射场、火箭、卫星制造商、成功失败、卫星数。
- 总览页“最近发射”和“最近发射卫星”标题保持不变，但内容实际展示全部发射/卫星，并使用独立滚动区域。
- 火箭统计只统计火箭型号，不按 Y 号拆分；长征五号B只记录“长征五号B”，不要包含后面的“/远征二号”或 Y 号。

### 验证习惯
- 后端改动优先跑 `uv run pytest`，必要时可只跑相关测试。
- 前端改动跑 `cd frontend && npm run lint && npm run build`。
- 发布相关改动跑 `uv build`，并检查 wheel 中是否包含 `gw/web/static/index.html`、JS、CSS、favicon。

## Git 提交规范

提交信息遵循 `<type>(<scope>): <subject>` 格式，类型与范围如下：

### Type（类型）
| 类型 | 含义 |
|------|------|
| feat | 新功能 |
| fix | 修复 bug |
| docs | 仅文档改动 |
| style | 代码格式（不影响功能的空格、分号等） |
| refactor | 重构（既不是新功能也不是修复） |
| perf | 性能优化 |
| test | 添加或修改测试 |
| chore | 构建、工具、依赖等杂项 |
| ci | CI/CD 配置改动 |

### Scope（范围）
| 范围 | 说明 |
|------|------|
| backend | 后端通用 |
| frontend | 前端通用 |
| scraper | 爬虫/数据采集 |
| db | 数据库模型/迁移 |
| web | Web 服务/API |
| daemon | 守护进程 |
| config | 配置文件/加载逻辑 |
| docs | 文档 |
| build | 打包/构建 |
| deps | 依赖更新 |

### 示例
```
feat(frontend): 增加发射统计页面
docs(readme): 更新安装说明
fix(backend): 修复 TLE 入库重复问题
refactor(scraper): 优化数据解析逻辑
```
